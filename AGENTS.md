## Product Context

Autoppia Studio is the studio/backend for configuring agents, connectors, tools,
benchmarks, task harvesting, trajectory judging, and skill creation.

The runtime naming preferred by the user is:

- `AgentConfig`: the definition/configuration of an agent.
- `AgentRuntime`: the deployed runtime that executes `/step` using an
  `AgentConfig`.
- Avoid introducing new `operator` naming. Existing legacy names may remain only
  for compatibility, but new code and UI text should use `agent`.

The runtime service itself may be external. This repo creates and passes the
agent configuration; the external runtime decides model execution unless this
repo explicitly implements it.

## Core Vocabulary

Use these meanings consistently:

- `Task`: the prompt plus everything needed to evaluate one thing. This includes
  start URL/state, seed/fixtures, success criteria, evaluator config, and any
  task metadata. A task is not a solution.
- `Benchmark`: a group of tasks, usually with shared metadata and run config.
- `Trajectory`: a concrete sequence of actions that attempts to solve a task.
  This is a solution candidate generated by a harvester or runtime.
- `Skill`: an approved trajectory promoted into a callable capability that the
  agent can use later at runtime.

Expected flow:

```text
Benchmark -> Tasks -> Harvester receives Task -> Candidate Trajectory
-> Judge evaluates Task + Trajectory -> approved Trajectory -> Skill
-> AgentRuntime can call Skill
```

Do not model pending work as rows in `trajectories`. Pending work belongs in
benchmark/task records. `trajectories` should represent generated solutions or
action traces.

## Data Model

Important Mongo collections in `backend/app/database.py`:

- `benchmarks_collection`: benchmark metadata.
- `benchmark_tasks_collection`: benchmark tasks.
- `trajectories_collection`: generated trajectories/action traces.
- skills/capabilities collections: approved callable skills.
- agents/connectors/tools/credentials collections: studio runtime config.

Legacy compatibility exists for old pending harvest rows in
`trajectories_collection` with `needs_harvest`, but new code should use
`benchmark_tasks_collection`.

Secrets and auth data should not be printed. If `.env` must be inspected, filter
or redact values.

## Harvesters

Main abstraction: `backend/app/services/agent_harvesters.py`.

Use:

- `AgentHarvester.harvest(agent_config)` for batch harvests.
- `AgentHarvester.harvest_task(agent_config, HarvestTask)` as the primitive.
- `harvest_one(...)` only exists as a legacy alias.

`HarvestTask` wraps a task record. Harvesters should receive tasks, not
trajectories.

Claude harvester timeout default is intentionally higher:

- File: `backend/app/harvester/claude_cli.py`
- Env var: `AUTOMATA_HARVESTER_TIMEOUT_SECONDS`
- Default: `600`

Full harvester benchmarks can be slow and can spend LLM/browser resources. Do
not rerun the full 14-web benchmark unless the user asks for it. Prefer small
smokes such as one project and one task.

## Judges

Main abstraction: `backend/app/services/trajectory_judges.py`.

`TrajectoryJudgeContext` is the shared input for all judge implementations. It
should carry everything a stateful evaluator or LLM judge needs:

- agent config
- normalized task spec
- trajectory/actions
- harvester result
- metadata
- artifacts
- evaluator config

Available judge implementations include:

- `rules`
- `hardcoded_html`
- `llm`
- `iwa`

The IWA judge should use the real IWA `AsyncStatefulEvaluator`; it is not just a
metadata/confidence placeholder. It should replay actions and score the final
state using IWA task definitions/golden tests when available.

The LLM judge should receive the same complete `TrajectoryJudgeContext`
serialized for judging, including artifacts when present.

## API And Runtime

Prefer `/step` naming over `/act`. If legacy `/act` remains, it should be
compatibility only.

When reviewing runtime behavior, inspect exactly what `/step` receives. The
important question is whether `AgentConfig` is included and whether credentials,
tools, skills, connectors, and knowledge are available in the shape the runtime
expects.

Skills should be visible in the UI/chat timeline when used, so users can
distinguish "agent manually operated browser" from "agent called an approved
skill".
